# DECISIONS

Decision | Reason | Impact | Date
Use `YoutubeBOTUpload-master` as the effective project root | Workspace outer folder only wraps the actual solution and code modules | All project memory, AGENTS rules, and further analysis should reference this directory | 2026-03-23
Do not use multi-agent for this task | Repo instructions encourage evaluation, but current higher-level tooling policy only allows subagents when the user explicitly asks for delegation | Analysis is performed in the main thread to stay compliant | 2026-03-23
If rebuilding from scratch, prefer `React + FastAPI + Postgres + Redis + Python worker` over desktop-bot architecture | The product is primarily an internal media automation dashboard with file processing and background jobs; Python fits the worker-heavy, FFmpeg/OAuth/API integration shape well | Future planning should optimize for stateless web services on VPS and eliminate Windows/Selenium as the core runtime dependency | 2026-03-23
Prefer stable, attributable business egress IP design over rotating/anonymous proxy design for YouTube uploads | Hiding or rotating IPs to avoid trust systems creates policy and account-risk; legitimate multi-channel operations are better served by consistent network identity, auditability, and quota/compliance planning | Network architecture should center on static IPs, tenant isolation where justified, and API-compliant operations rather than evasion tactics | 2026-03-23
Prefer a hybrid architecture when render outputs are very large: cloud control plane + local high-storage render worker | Large FFmpeg workloads are storage- and scratch-disk-heavy; keeping them on a strong local machine avoids costly VPS storage and I/O bottlenecks while preserving centralized job orchestration | Future design should treat VPS as control plane and local/NVMe worker as data plane, with outbound agent/tunnel connectivity but no large media transfer through the tunnel | 2026-03-24
Preserve old frontend information architecture and workflow concepts, but rebuild the visual system and interaction layer from scratch | Existing admin/user screens already reveal the right entities and job flow, but the current Bootstrap 4/Stisla/jQuery presentation is dated and inconsistent | The new frontend should reuse navigation structure, field semantics, and status models while replacing layout, components, and UI technology | 2026-03-24
For the current target architecture, use one central VPS as control plane and multiple worker VPS nodes for processing/upload instead of the earlier local-worker assumption | The product now needs centralized web/API control over a larger bot fleet, with manager-based distribution of worker capacity to users | Planning, IA, and backend models should include worker pools, manager-to-bot assignment, and remote worker orchestration as first-class concepts | 2026-03-24
Allow end users to connect YouTube OAuth accounts directly in the new UX, while admin remains responsible for account creation and bot allocation | This removes an unnecessary admin bottleneck in daily operations and matches the updated ownership model where users operate within assigned worker capacity | User-facing UI and backend permissions should support self-service channel connection inside the scope of assigned resources | 2026-03-24
